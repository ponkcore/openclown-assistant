# PRD-003: Tracking Modalities Expansion

## 1. Problem Statement
PRD-001@0.2.0 shipped a single-modality assistant: KBJU (calories, protein, fat, carbs) tracking via voice / text / photo inputs. The Product Owner's stated long-horizon vision ("персональный ассистент, который помогает графики строить жизни, учёбу планировать") and the v0.2 roadmap require a wider modality surface so the assistant can stand on more than one signal per user. Four user-life modalities have been authorised by the Product Owner for the v0.2 modality-expansion epic: water (hydration), sleep, workouts (exercise), and mood / energy. Without these, downstream epics (proactive coaching in the next PRD; calendar + web visualization in the PRD after that; life-manager modules later) cannot offer cross-modality insights or adaptive UX with anything other than KBJU as the single trackable input. PRD-003 establishes the modality-tracking layer, but does NOT introduce any recommendation, nudge, or cross-modality coaching behaviour — those live in the proactive-coaching PRD that follows. PRD-003 also establishes the per-modality on/off granularity that the Product Owner explicitly requested ("если пользователь отслживает всё, то тогда в одной сводке, а если нет - то нет; тут надо подстраиваться под юзера"), so summaries adapt to each user's active modality set without becoming a single rigid form. PRD-003 ships AFTER PRD-002@0.2.1 ratifies and the corresponding observability + scale-readiness architecture spec lands, so every new modality inherits continuous tenant-isolation breach detection (PRD-002@0.2.1 G1), automated model-stall detection (PRD-002@0.2.1 G2), and the scale-ready Telegram allowlist with documented code-path audit (PRD-002@0.2.1 G4) without ever needing to re-justify those guarantees.

## 2. Goals (SMART)
- **G1 — Water / hydration tracking enabled.** From v0.2 modality-deploy onward, allowlisted users can log a hydration event by voice, free-form text, OR a Telegram inline keyboard preset (one of the established quick-volumes documented in §5 US-1) and receive a confirmation reply within the PRD-001@0.2.0 §7 latency envelopes (voice ≤8 s p95, text ≤5 s p95). Each hydration event records volume in millilitres and timestamp at ≤1-second precision, persists per-user in the same multi-tenant storage layer used by PRD-001@0.2.0 §5 US-2 / US-3 / US-4, and is included in PRD-001@0.2.0 §5 US-7 daily / weekly / monthly summaries when the user has the water modality enabled. Coverage = 100 % of allowlisted users that have water modality ON. Closes Q-MOD-1 / Q-MOD-3 (PO ratified BP defaults: multi-input water logging).
- **G2 — Sleep tracking enabled.** From v0.2 modality-deploy onward, allowlisted users can log a sleep event by either (a) a single morning report in voice or free-form text describing total duration ("спал 7 часов", "spal 7 h"), OR (b) a paired evening "лёг" / morning "встал" event sequence from which the assistant derives total duration; both paths produce a single sleep-record object with start timestamp, end timestamp, and derived duration (≤1-second precision; timezone honoured per the user's onboarding-locked timezone). Each sleep record persists per-user, is included in PRD-001@0.2.0 §5 US-7 summaries when the user has the sleep modality enabled, and is rejected with an actionable correction prompt if duration < 30 min or > 24 h (sanity floor / ceiling; soft-warn, not hard-fail). Coverage = 100 % of allowlisted users that have sleep modality ON. Closes Q-MOD-1 / Q-MOD-3 (PO ratified BP defaults: dual-path sleep logging).
- **G3 — Workout / exercise tracking enabled.** From v0.2 modality-deploy onward, allowlisted users can log a workout event by free-form text ("бегал 30 мин 5 км"), voice ("сегодня жал 80 кило 5 на 5"), OR photo (workout-record screenshot from a phone tracker app, or an action photo of equipment / outdoors); each event records workout type (free-form taxonomy, normalised to a canonical label by the assistant), duration in minutes (where derivable), distance in kilometres (where derivable), and intensity level (low / medium / high) where derivable. Each event persists per-user, is included in PRD-001@0.2.0 §5 US-7 summaries when the user has the workout modality enabled. Coverage = 100 % of allowlisted users that have workout modality ON; recognition success-rate target ≥80 % for workout-type extraction on a manually-curated 50-event golden set, ratified by the Product Owner before sign-off. Closes Q-MOD-1 / Q-MOD-3 (PO ratified BP defaults: text + voice + photo workout logging).
- **G4 — Mood / energy tracking enabled.** From v0.2 modality-deploy onward, allowlisted users can log a mood / energy event by either (a) a 1–10 numeric scale via Telegram inline keyboard or free-form text ("настроение 7", "энергия 4"), with an OPTIONAL free-form text comment of ≤280 characters; OR (b) a free-form descriptive text alone (the assistant maps it to the 1–10 scale and replies with the inferred numeric value for user confirmation). Each event records the integer 1–10 score, the optional comment (subject to PII redaction at emit per ARCH-001@0.4.0 §10.7), and the timestamp at ≤1-second precision. Each event persists per-user, is included in PRD-001@0.2.0 §5 US-7 summaries when the user has the mood modality enabled. Coverage = 100 % of allowlisted users that have mood modality ON. Closes Q-MOD-1 / Q-MOD-3 (PO ratified BP defaults: numeric scale + optional text comment, with free-text-to-scale inference fallback).
- **G5 — Per-modality on / off settings.** From v0.2 modality-deploy onward, every user has independent on / off toggles for each of the four PRD-003 modalities (water, sleep, workout, mood); KBJU from PRD-001@0.2.0 §5 US-2 / US-3 / US-4 is always-on (KBJU is the v0.1 anchor and cannot be disabled). All four PRD-003 modalities default ON for new users; existing v0.1 users (Product Owner + partner) inherit ON-on-deploy unless they explicitly disable post-deploy. Toggling a modality off MUST take effect within ≤30 seconds (no further events accepted for that modality until re-enabled), MUST NOT delete any historical data, and MUST cause the modality's section to disappear from subsequent summaries within the same window. Toggling a modality back on resumes acceptance and resumes summary inclusion for events from that re-enable moment forward. Coverage = 100 % of allowlisted users. Closes Q-MOD-2 (PO direction: ship the best solution, observe user reaction, iterate; BP recommends granular-on-by-default).
- **G6 — Adaptive summary integration.** From v0.2 modality-deploy onward, the PRD-001@0.2.0 §5 US-7 daily / weekly / monthly summary text template adapts to each user's active modality set: KBJU section always present; water / sleep / workout / mood sections appear ONLY when that modality is currently ON for that user AND has at least one logged event in the summary window. Sections suppressed for OFF modalities or zero-event modalities. Section ordering: KBJU → water → sleep → workout → mood (deterministic, not user-rearrangeable in this PRD; future personalization PRD may revisit). Adaptive section selection is config-driven (not learning-based; learning belongs in the proactive-coaching PRD per §3 NG2). Coverage = 100 % of generated summaries; correctness = summary section set MUST exactly match active-modality set for every generated summary in a rolling-7-day audit window. Closes Q-MOD-4 (PO ratified: "если пользователь отслеживает всё, то тогда в одной сводке, а если нет - то нет").

## 3. Non-Goals (explicitly NOT in this epic)
- **NG1.** No proactive nudges, prompting, or reminder cadence around any of the four new modalities (no "ты сегодня выпил мало воды", no "три дня нет тренировок — попробуй 20-мин ходьбу", no "недосып третий день, проверь распорядок"). Deferred to the proactive-coaching PRD that follows PRD-003.
- **NG2.** No cross-modality recommendations, correlations, or insights (no "ты плохо спал, выпей больше воды", no "энергия низкая после поздних тренировок", no "паттерн: настроение падает в дни без зарядки"). Deferred to the proactive-coaching PRD that follows PRD-003 (PO ratified Q-MOD-5: "только трекинг в PRD-003").
- **NG3.** No food coaching changes from PRD-001@0.2.0 §3 NG6 (no vitamin / glycemic-index / supplement / hydration recommendations introduced by PRD-003; hydration tracking is logging-only, not coaching). Food coaching expansion is deferred to the proactive-coaching PRD per the boundary defined in PRD-002@0.2.1 §10.
- **NG4.** No external sleep / fitness / health tracker integrations (Apple Health, Google Fit, Oura, Whoop, Garmin, Fitbit, Strava, MyFitnessPal, etc.). Modality data in v0.2 is exclusively user-entered through Telegram. External integrations are explicitly out of v0.2 scope and may be considered no earlier than the calendar-and-web-view PRD (which itself addresses external Google / Yandex calendar integration as the first instance of external-system sync).
- **NG5.** No new user-facing channel beyond Telegram. Per PRD-001@0.2.0 §7 + PRD-002@0.2.1 §7 (Product-Owner-locked), Telegram remains the primary product channel; web visualization is deferred to the calendar-and-web-view PRD; no SMS, no email, no Apple-Watch, no standalone iOS / Android app.
- **NG6.** No replacement of PRD-001@0.2.0 §5 user stories US-1 through US-9. The KBJU voice / text / photo logging surface, the daily / weekly / monthly summaries, the history-mutation flow, the right-to-delete flow, and the multi-tenant data model remain in effect; PRD-003 only adds four parallel modalities alongside KBJU and adds per-modality settings + adaptive summary integration. KBJU stays always-on (G5) and is never gated by a modality toggle.
- **NG7.** No medical or clinical advice across any of the four new modalities. PRD-001@0.2.0 §3 NG7 is reaffirmed without modification. PRD-003 makes no statement about diagnoses, conditions, supplements, sleep disorders, exercise injuries, or mood disorders, on-request or unprompted. Sleep duration sanity floor / ceiling (G2) is data-quality plumbing, not a clinical judgement.
- **NG8.** No subscription, billing, payment, promo-code, or referral functionality. Extends PRD-001@0.2.0 §3 NG4 + PRD-002@0.2.1 §3 NG6 indefinitely; the Product Owner has confirmed monetization stays out of scope for the entire v0.2 cycle.
- **NG9.** No per-user assistant personality customization (no preset picker, no tone-of-voice options). Preserves PRD-001@0.2.0 §3 NG10 + PRD-002@0.2.1 §3 NG5; the proactive-coaching PRD will revisit only the narrow adaptive-UX subset.
- **NG10.** No SDLC pipeline token / cost / role-attribution telemetry — preserved from PRD-002@0.2.1 §3 NG9. Tracking Business Planner / Architect / Executor / reviewer consumption is out of scope for the entire v0.2 cycle.
- **NG11.** No retroactive backfill of pre-deploy modality data. Users cannot enter "вчера выпил 2 литра воды" or "на прошлой неделе бегал 3 раза по 5 км" through any past-dated input path; events timestamp at the moment of logging only. Historical entry by past-date is deferred to a future PRD if Product Owner ratifies a need.

Not customers of the product; first-class consumers of PRD-003 output for coordination decisions.

## 5. User Stories & Acceptance Criteria

### US-1: Water / hydration logging (G1)
**As an allowlisted user with water modality ON,** I want to log a hydration event by voice, free-form text, OR a Telegram inline-keyboard quick-volume preset, **so that** my water intake counts toward my daily / weekly / monthly summaries without me needing to remember a specific command syntax.
**Acceptance:**
- [ ] Given water modality is ON, when I send a voice message saying a phrase containing a hydration reference + a volume in litres or millilitres ("выпил 200 миллилитров", "стакан воды"), then a hydration event is persisted with my user_id, the parsed volume in millilitres, and the message timestamp at ≤1-second precision, and I receive a confirmation reply within the PRD-001@0.2.0 §7 voice-latency envelope (≤8 s p95).
- [ ] Given water modality is ON, when I send a free-form text message containing a hydration reference + a volume ("200ml воды", "пол-литра"), then the event is persisted equivalently and I receive a confirmation reply within the PRD-001@0.2.0 §7 text-latency envelope (≤5 s p95).
- [ ] Given water modality is ON, when I press a quick-volume inline-keyboard button (the bot exposes at least three presets — small / medium / large, with concrete millilitre values ratified by the Product Owner before sign-off), then a hydration event is persisted equivalently and I receive a confirmation reply within the PRD-001@0.2.0 §7 text-latency envelope.
- [ ] Given water modality is OFF, when I send any of the above inputs, then the message is treated as either a KBJU input (if it matches a meal pattern) or a no-op friendly reply telling me water modality is currently disabled in my settings; no hydration event is persisted.
- [ ] Given a hydration event is persisted, when I later run `/forget_me` per PRD-001@0.2.0 §5 US-8, then that event is hard-deleted in the same transaction boundary as my KBJU records (right-to-delete extension; consistent with PRD-002@0.2.1 §5 US-5 telemetry-trace deletion).

### US-2: Sleep tracking (G2)
**As an allowlisted user with sleep modality ON,** I want to log my sleep either as a single morning duration report OR as a paired evening / morning event sequence, **so that** I can track my sleep without being forced into one rigid input pattern.
**Acceptance:**
- [ ] Given sleep modality is ON, when I send a morning voice or text message containing a sleep duration ("спал 7 часов", "слип 6.5 ч", "спал 7h"), then a sleep record is persisted with my user_id, the parsed total duration, the inferred end timestamp (≈ message receipt timestamp), the inferred start timestamp (= end − duration), and the same modality-event metadata as US-1.
- [ ] Given sleep modality is ON, when I send an evening "лёг" / "иду спать" event followed by a morning "встал" / "vstal" event in the same Telegram chat within 24 hours, then a single sleep record is created with the evening event's timestamp as start, the morning event's timestamp as end, and the difference as duration; if a "лёг" event is followed by another "лёг" event without an intervening "встал", the older one is invalidated and a friendly reply asks the user to clarify.
- [ ] Given a sleep record with start timestamp on calendar day D and end timestamp on calendar day D+1 (i.e. the sleep span crosses midnight in the user's onboarding-locked timezone), when daily / weekly / monthly summaries are generated, then the record is attributed to and included in the calendar day of the END (waking) timestamp only — i.e. the D+1 daily summary — and is NOT also included in the D daily summary; weekly / monthly window membership is determined identically by the END timestamp's calendar day.
- [ ] Given a sleep record has a derived duration ≤ 4 hours, the record is persisted as a separate nap-class sleep record and is NOT auto-merged with any other sleep record of the same calendar day; multiple naps in the same day are each persisted as independent records, each attributed to the calendar day of its own end timestamp per the midnight-spanning rule above.
- [ ] Given sleep modality is ON, when a parsed duration is < 30 minutes or > 24 hours, then the record is NOT persisted; instead a friendly Telegram reply asks the user to confirm whether the duration is correct, and the user can confirm-as-is (then persist) or correct (then re-parse).
- [ ] Given sleep modality is OFF, when I send any of the above inputs, then the message is treated as either a KBJU input (if it matches a meal pattern) or a no-op friendly reply telling me sleep modality is currently disabled in my settings; no sleep record is persisted.
- [ ] Given a sleep record is persisted, when I later run `/forget_me`, then that record is hard-deleted in the same transaction boundary as my KBJU records.

### US-3: Workout / exercise tracking (G3)
**As an allowlisted user with workout modality ON,** I want to log a workout by free-form text, voice, OR photo, **so that** I can capture both quick context-rich messages ("бегал 30 мин 5 км") and tracker screenshots (photo of a fitness app's session screen) without learning a domain-specific log format.
**Acceptance:**
- [ ] Given workout modality is ON, when I send a free-form text message describing an exercise event (running / cycling / strength training / yoga / etc.) with at least one of {duration, distance, repetitions × sets, intensity descriptor}, then a workout event is persisted with my user_id, the canonical workout-type label (drawn from a free-form-but-normalised taxonomy ratified by the Product Owner before sign-off), the parsed duration in minutes if derivable, the parsed distance in kilometres if derivable, the inferred intensity level (low / medium / high) if derivable, and the message timestamp.
- [ ] Given workout modality is ON, when I send a free-form text message describing a workout type but lacking ANY of {duration, distance, repetitions × sets, intensity descriptor} (zero quantifiable fields parseable from the message), then the bot replies with a friendly clarifying question asking me for at least one quantifiable detail (duration, distance, reps × sets, or intensity), AND NO workout event is persisted until I reply with the missing field; the message MUST NOT silently fall through to KBJU parsing and MUST NOT be dropped without a reply.
- [ ] Given workout modality is ON, when I send a voice message describing the same, then the event is persisted equivalently after speech-to-text + the same parsing pipeline.
- [ ] Given workout modality is ON, when I send a photo of a fitness-tracker workout-summary screen, then the bot extracts workout-type / duration / distance / intensity from the photo at the same fidelity as text input where possible; if extraction fails on any required field, the bot replies with a friendly clarifying question rather than persisting an incomplete record.
- [ ] Given a manually-curated 50-event golden set covering the most common workout types (running, walking, cycling, strength training, yoga, swimming, hiking) the workout-type extraction success rate is ≥80 %; the golden-set composition is ratified by the Product Owner before sign-off.
- [ ] Given workout modality is OFF, when I send any of the above inputs, then the message is treated as either a KBJU input (if it matches a meal pattern) or a no-op friendly reply; no workout event is persisted.
- [ ] Given a workout event is persisted, when I later run `/forget_me`, then that event is hard-deleted in the same transaction boundary as my KBJU records.

### US-4: Mood / energy tracking (G4)
**As an allowlisted user with mood modality ON,** I want to log my mood or energy as a 1–10 numeric value (with an optional free-text comment), OR as a free-form descriptive text from which the assistant infers the numeric value, **so that** I can capture how I feel today in either ≤2 seconds (numeric only) or with as much context as I choose.
**Acceptance:**
- [ ] Given mood modality is ON, when I press a Telegram inline-keyboard 1–10 button or send text matching a numeric-1–10 pattern ("настроение 7", "энергия 4", "mood 8"), then a mood event is persisted with my user_id, the integer 1–10 score, an empty comment field, and the message timestamp.
- [ ] Given mood modality is ON, when I send a numeric score followed by a free-form comment ("настроение 7 — устал но в целом норм"), then the same record is persisted with the integer score AND the comment text (≤280 characters; longer comments are accepted but truncated at persistence with a friendly notice; comment is subject to ARCH-001@0.4.0 §10.7 redaction at emit boundary).
- [ ] Given mood modality is ON, when I send a free-form descriptive text without a numeric score ("чувствую себя дерьмово", "сегодня просто отлично"), then the assistant infers the 1–10 score, replies with the inferred score for me to confirm or correct, and persists the record only after my confirmation reply.
- [ ] Given a pending mood free-text inference is awaiting my confirmation or correction, when 5 minutes elapse without my explicit confirmation or correction, then the pending inference is discarded silently AND my next message is processed as a new independent input (NOT auto-treated as confirmation of the discarded inference); a fresh free-text mood input restarts the inference + confirmation flow with a new 5-minute window.
- [ ] Given mood modality is OFF, when I send any of the above inputs, then the message is treated as either a KBJU input (if it matches a meal pattern) or a no-op friendly reply; no mood record is persisted.
- [ ] Given a mood event is persisted, when I later run `/forget_me`, then that event AND its comment text are hard-deleted in the same transaction boundary as my KBJU records.

### US-5: Per-modality on / off settings (G5)
**As an allowlisted user,** I want independent on / off toggles for each of the four PRD-003 modalities, **so that** I can track only what I care about without seeing summary sections for modalities I never use.
**Acceptance:**
- [ ] Given a `/settings` (or equivalent menu entry; exact label ratified by the Product Owner before sign-off) Telegram command, when I invoke it, then I see a settings surface with four toggles labelled in Russian for water / sleep / workout / mood, each showing its current ON / OFF state.
- [ ] Given the settings surface, when I toggle a modality OFF, then within ≤30 seconds of the toggle commit any new event input for that modality is rejected with a friendly reply (per US-1 / US-2 / US-3 / US-4 OFF acceptance bullet) and the next generated summary suppresses the modality's section per US-6.
- [ ] Given the settings surface, when I toggle a modality back ON, then within ≤30 seconds new event inputs for that modality are accepted and the next generated summary resumes including the modality's section per US-6.
- [ ] Toggling a modality OFF MUST NOT delete any historical events for that modality; pre-toggle data is preserved and remains subject to PRD-001@0.2.0 §5 US-8 right-to-delete.
- [ ] All four modalities default ON for new users; existing v0.1 users (Product Owner + partner) inherit all-ON on the modality-deploy moment.
- [ ] KBJU is NOT a toggleable modality and does NOT appear in the settings surface (per §3 NG6 + G5 always-on KBJU rule).

### US-6: Adaptive summary integration (G6)
**As an allowlisted user,** I want each daily / weekly / monthly summary to include only the modalities I currently have ON (and that have at least one event in the summary window), **so that** my summary is concise and reflective of my actual tracking habits, not padded with empty sections.
**Acceptance:**
- [ ] Given a generated summary, the KBJU section is always present (per §3 NG6 + G5 always-on KBJU rule) regardless of any modality toggle.
- [ ] Given water / sleep / workout / mood modality X is currently ON for me, when a summary is generated and at least one event for modality X exists in the summary window, then the summary INCLUDES a section for modality X.
- [ ] Given modality X is currently OFF for me, when a summary is generated, then the summary OMITS the section for modality X regardless of any historical events for modality X.
- [ ] Given modality X is currently ON but ZERO events exist in the summary window, then the summary OMITS the section for modality X (zero-event suppression to keep summaries concise; events from prior windows do not count).
- [ ] Section ordering in any summary is deterministic: KBJU → water → sleep → workout → mood.
- [ ] A rolling-7-day audit of summary correctness MUST show 100 % match between active-modality set and summary-section set (i.e., zero summaries with a modality section the user disabled, zero summaries missing a modality section the user has enabled with events). Audit method ratified by the Product Owner and the Architect during the PRD-003 successor architecture spec design.

### US-7: Privacy-preserving modality telemetry (cross-cutting)
**As any user (Product Owner, partner, future user),** I want my modality data to remain redacted from every alert and structured-log channel that PRD-003 introduces, **so that** the new modalities do not erode the data-isolation guarantee established by PRD-001@0.2.0 §6 K4 + PRD-002@0.2.1 §5 US-5.
**Acceptance:**
- [ ] Given any C10-emitted telemetry event introduced by PRD-003 (modality-specific event counters, per-modality recognition success-rate metrics, settings-toggle audit rows), no allowlisted forbidden field (`meal_text`, mood-comment text, workout free-form description text, sleep-text-input transcript, raw audio, raw photo bytes, usernames, raw transcripts, raw prompts, raw media, provider keys, provider responses; per ARCH-001@0.4.0 §10.7 emit-boundary redaction policy and §8.1 allowlist) appears in alert or structured-log output channels.
- [ ] A sample audit of N ≥ 100 PRD-003 telemetry events drawn at random from a rolling-7-day window shows 100 % redaction compliance.
- [ ] Any new telemetry event added by PRD-003 must declare which redaction-allowlist entries it asserts and is rejected at the emit boundary if it carries any non-allowlisted field.
- [ ] When a user invokes `/forget_me` per PRD-001@0.2.0 §5 US-8, the right-to-delete transaction also hard-deletes that user's PRD-003 modality records (water / sleep / workout / mood events + per-event comment text + settings-toggle audit rows tagged with that `user_id`) in the same transaction boundary as the existing user-scoped data.

## 7. Technical Envelope (constraints Architect must respect)

## 8. Risks & Mitigations
| Risk | Impact | Likelihood | Mitigation |
|---|---|---|---|
| **R1 — modality-input ambiguity: a message like "выпил пол-литра" could match either a water-tracking pattern OR a KBJU drink pattern; the wrong path is taken.** | medium | medium–high | The Architect defines an explicit modality-priority resolution at the parser layer (default order: KBJU → water → sleep → workout → mood for ambiguous inputs, ratified by PO before sign-off); ambiguous inputs trigger a friendly clarifying reply rather than silent best-guess persistence; rolling-30-day modality-misclassification rate tracked as informational telemetry. |
| **R2 — sleep tracking edge cases: timezone shifts (travel), naps (multiple short events per day), fragmented sleep (split between 2 windows), DST transitions.** | medium | medium | The Architect addresses each edge case explicitly in the PRD-003 successor architecture spec (timezone honoured per onboarding-locked timezone; nap handling is normative per §5 US-2 AC — ≤4 h derived duration = separate nap-class record with no auto-merge; midnight-spanning attribution is normative per §5 US-2 AC — record attributed to the calendar day of the END timestamp only; fragmented sleep treated as two records the user can manually link in a future PRD; DST transitions calculated on a per-record basis). Sanity floor / ceiling (G2) catches obvious data-quality issues; clarifying replies rather than silent rejection. |
| **R3 — workout-photo recognition limited by what the existing photo-extraction adapter (per ARCH-001@0.4.0 §3.8) can read; fitness-tracker UIs vary widely.** | medium | medium–high | G3 / K2 specifies an 80 % accuracy target on a PO-ratified 50-event golden set rather than open-ended "all fitness apps work"; the golden set composition explicitly bounds the success criterion; below-threshold extraction falls back to clarifying-reply rather than silent best-guess persistence; ratification by PO required before sign-off. |
| **R4 — mood-comment PII leak: free-form mood comments could contain sensitive personal context (relationship, health, other people's names) that leaks into operator dashboards or summary-generation prompts.** | critical | medium | The Architect adds mood-comment text to the ARCH-001@0.4.0 §8.1 redaction allowlist explicitly via a successor TKT-mod-redact ticket; US-7 acceptance includes a mood-comment-specific 100-event audit (K4); summary-generation prompts MUST NOT inline mood-comment text in any operator-readable channel; user-facing summary may surface the score (1–10) but treats the comment as user-only-visible content. |
| **R5 — per-modality settings race condition: a toggle commit happens during a summary generation; the summary uses the pre-toggle state instead of the post-toggle state, causing a transient mismatch.** | low | medium | The Architect defines a serialisation contract for settings reads vs summary generation (e.g. summary generation snapshots the user's settings at the start of generation; the snapshot timestamp is recorded; toggles after that timestamp do not affect the in-progress summary). G6 / K6 audit treats the snapshot timestamp as the comparison point, not the summary-emit timestamp. |
| **R6 — adaptive summary template combinatorial explosion: 4 toggleable modalities + KBJU = 32 possible section sets; the assistant generates one of 32 templates per summary; testing surface explodes.** | medium | medium | Section ordering is deterministic (G6 acceptance), so the template is parameterised by the active set rather than 32 distinct templates; the PRD-003 successor architecture spec defines a single parametric template; G6 / K6 audit covers the parametric template's correctness rather than enumerating all 32 cases. |
| **R7 (weakest assumption A1) — modality-input ambiguity (R1) is solvable through deterministic rules alone (priority order + clarifying replies on ambiguity).** | medium | medium | The Architect's R1 mitigation is deterministic-rule-based; if production telemetry shows a modality-misclassification rate >5 % on rolling-30-day window, the Architect escalates to the Product Owner for an LLM-backed disambiguation step (fallback-only, not default; treated as a future PRD if needed). |
| **R8 (weakest assumption A2) — adaptive summary section selection at generation time (G6) does not require a new persistent settings-snapshot store; the user's current settings can be read live at summary-emit time without race conditions.** | low | medium | If R5 mitigation (in-flight snapshot) reveals that live read is insufficient, the Architect adds a settings-snapshot-at-emit table without changing the PRD scope; the storage is a tactical implementation choice, not a goal. |
| **R9 (weakest assumption A3) — workout-type extraction (G3) achieves ≥80 % accuracy on the 50-event golden set on the first the PRD-003 successor architecture spec implementation iteration.** | medium–high | medium | G3 / K2 specifies the gating condition; if the first iteration falls short, the Architect iterates (extraction-prompt tuning, taxonomy expansion) before sign-off; if iterations stall below 70 %, the Architect escalates to PO with options (lower accuracy threshold, narrow taxonomy, deferred photo path); R3 mitigation applies in parallel. |
