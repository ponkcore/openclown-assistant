# PRD-002: Observability and Scale Readiness

## 1. Problem Statement
PRD-001@0.2.0 closed end-to-end (ARCH-001@0.4.0 + 15 implementation tickets merged) but no real-user 30-day pilot has yet started, so all PRD-001@0.2.0 §6 K1–K7 baselines remain `n/a`. Two structural reliability gaps surfaced from the implementation pilots: (1) the tenant-isolation guarantee from PRD-001@0.2.0 §3 NG / §6 K4 is verified only by an end-of-pilot audit, not continuously; (2) model-stall recovery for the assistant's runtime LLM calls (context-window exhaustion) was caught manually mid-pilot, with no automated detection. v0.2 also opens the user base from the 2-pilot population locked in PRD-001@0.2.0 §3 NG3 to a scale-ready growth path (Product Owner → partner → friends → eventually thousands), so the existing static Telegram allowlist (locked in ARCH-001@0.4.0 §3.1) and the PRD-001@0.2.0 §7 single-VPS resource ceiling no longer fit. PRD-002 closes those two reliability gaps and establishes the scale-readiness foundation that the rest of the v0.2 roadmap (modalities expansion, proactive coaching + food guidance, calendar + web-view, life-manager modules) will be measured against; it ships before the modality and coaching epics so the next PRDs have continuous breach detection, automated stall recovery, and a config-driven user-access path to build on. Per-role / per-call SDLC pipeline token-cost telemetry (formerly considered for this PRD) was explicitly de-scoped by the Product Owner ("на этапе разработки плевать на стоимость; мы вольны тратить столько, сколько потребуется для классного продукта; команду разработки точно не нужно отслеживать"); see §3 NG9 and §9 OQ-1.

## 2. Goals (SMART)
- **G1 — Continuous tenant-isolation breach detection.** From v0.2 deploy onward, every user-scoped storage operation that references a `user_id` other than the requesting principal emits a structured breach event within ≤5 minutes p95 of the operation, and a Telegram alert reaches the Product Owner within an additional ≤5 minutes. Over any rolling 30-day window the continuous detector must catch every breach the PRD-001@0.2.0 §6 K4 end-of-window audit catches (zero false-negatives). Measured by synthetic breach injection + structured-log timestamp diff. Promotes PRD-001@0.2.0 §6 K4 from end-of-pilot audit to continuous detection.
- **G2 — Automated model-stall detection.** From v0.2 deploy onward, any routed-model call (through the LLM-router layer or its direct-provider failover path) that produces no token output for ≥120 seconds (default; per-role configurable) emits a stall event within ≤15 seconds of the threshold being crossed, carrying enough context (call identifier, role, elapsed time, prompt-token count) for an operator to act on a takeover trigger. Coverage ≥95 % of routed call paths; synthetic stall scenarios at 120 s / 300 s / 600 s all detected within target. Closes the manual-only recovery pattern documented in.
- **G4 — Scale-ready Telegram allowlist with documented code-path audit.** From v0.2 deploy onward, the Telegram access list is config-driven (no code change required to add or remove users) and passes load tests at N = 2, N = 10, N = 100, N = 1 000, and N = 10 000 entries with allowlist-related per-update overhead ≤2 % of the user-facing latency budgets locked in PRD-001@0.2.0 §7 (voice ≤8 s p95, text ≤5 s p95, photo ≤12 s p95). Allowlist updates propagate from configuration to access enforcement within ≤30 seconds. Before any of these load-test gates is run, the Architect publishes a documented end-to-end allowlist code-path audit (every layer that performs an access check, every cache, every store round-trip, every linear-scan candidate) and the Reviewer signs off on it; the audit is a blocking gate for the load-tests, the load-tests are blocking gates for merge, and the merge is a blocking gate for advancing the user-count growth path past the last-passed gate. Removes the 2-tenant ceiling implied by PRD-001@0.2.0 §3 NG3.

## 3. Non-Goals (explicitly NOT in this epic)
- **NG1.** No new tracking modalities (no water / hydration, no sleep, no workout / exercise, no mood / energy logging). Deferred to the modalities-expansion PRD that follows PRD-002 in the v0.2 roadmap.
- **NG2.** No proactive nudges (missed-meal, under- or over-eating, macro-imbalance, pattern-triggered) and no adaptive-UX behaviour (assistant learning user preferences, adaptive questions, adjusting feature surfacing). Deferred to the proactive-coaching PRD.
- **NG3.** No food coaching beyond PRD-001@0.2.0 §3 NG6 (no vitamin / glycemic-index / supplement / hydration recommendations, on-request or unprompted). Deferred to the proactive-coaching PRD, which will define the new boundary against PRD-001@0.2.0 §3 NG7 (medical / clinical advice remains out of scope across the entire roadmap).
- **NG4.** No calendar integration of any kind (Google Calendar, Yandex Calendar, iCal, native calendars) and no web-based visualization layer (no read-only dashboards, no calendar-view web pages). Deferred to the calendar-and-web-view PRD.
- **NG5.** No per-user assistant personality customization (no preset picker, no tone-of-voice options). PRD-001@0.2.0 §3 NG10 is preserved for v0.2; the proactive-coaching PRD will revisit only the narrow adaptive-UX subset, not preset selection.
- **NG6.** No subscription, billing, payment, promo-code, or referral functionality. Extends PRD-001@0.2.0 §3 NG4 indefinitely; the Product Owner has confirmed monetization stays out of scope for the entire v0.2 cycle.
- **NG7.** No replacement of PRD-001@0.2.0 §5 user stories US-1 through US-9. The KBJU-only voice / text / photo logging surface, the daily / weekly / monthly summaries, the history mutation flow, the right-to-delete flow, and the multi-tenant data model remain in effect; PRD-002 only adds telemetry and scale around them, never replaces them.
- **NG8.** No medical or clinical advice. PRD-001@0.2.0 §3 NG7 is reaffirmed without modification; PRD-002 makes no statement about diagnoses, conditions, supplements, or pharmacology.
- **NG9.** No SDLC pipeline token / cost / role-attribution telemetry. Tracking Business Planner, Architect, Executor (default + parallel + specialist), and Reviewer (auto + manual) LLM consumption — formerly considered as a goal of this PRD — is explicitly out of scope for PRD-002 and the foreseeable v0.2 roadmap, per Product Owner direction ("на этапе разработки плевать на стоимость; команду разработки точно не нужно отслеживать"). Product-runtime cost telemetry for the assistant's own LLM calls remains tracked as a future open question (§9 OQ-1) but is also NOT a PRD-002 goal.

## 4. Target Users / Personas
- **P1 — Product Owner (primary v0.2 telemetry consumer).** Same individual as PRD-001@0.2.0 §4 P1. Receives Telegram alerts on tenant-breach detection and on model stalls; acts on takeover triggers when an LLM call stalls. Tech comfort high; expects structured numeric output, not narrative prose. Primary motivation: prevent silent data-isolation breach and silent stall on a runaway LLM call before the v0.2 modality / coaching / calendar epics ship.
- **P2 — Partner (secondary subject).** Same individual as PRD-001@0.2.0 §4 P2. No new user-facing surface from PRD-002; remains a passive subject of the same KBJU logging flows from PRD-001@0.2.0 §5. Continues to receive PRD-001@0.2.0 daily / weekly / monthly summaries unchanged. Telemetry collected about her interactions must be redacted per §7 of this PRD before reaching any alert.
- **P3 — Future user (3 to thousands).** Anyone added to the Telegram allowlist after the partner. Onboarded by the Product Owner out of band (PRD-001@0.2.0 §3 NG3 deprecated only for the user-count cap; no public sign-up, no marketplace listing). The growth path is Product Owner → partner → friends → wider population, with allowlist load-tested up to N = 10 000 before the next epic ships. Like P2, a passive subject of the same KBJU flows; gains no new user-facing features in PRD-002.
- **P4 — orchestrator and Architect (operational consumers).** Read the same alerts and ratification surfaces P1 reads to dispatch reviewers, detect mid-cycle Executor stalls, and ratify post-merge audits. They consume telemetry as inputs to coordination decisions (per the orchestrator session log and `docs/prompts/architect.md`); they are not customers of the product but they are first-class consumers of PRD-002 output.

## 5. User Stories & Acceptance Criteria

### US-1: Continuous tenant-isolation breach detection (G1)
**As the Product Owner,** I want every user-scoped storage operation that references a `user_id` other than the requesting principal to be detected and alerted within minutes, **so that** the tenant-isolation guarantee from PRD-001@0.2.0 §6 K4 is enforced continuously, not only at the end of a pilot window.
**Acceptance:**
- [ ] Given a user-scoped storage operation, when the operation references a `user_id` ≠ the requesting principal, then a structured breach event is emitted within ≤5 minutes p95 of the operation.
- [ ] Given a detected breach, when the event is emitted, then a Telegram alert reaches the Product Owner within an additional ≤5 minutes.
- [ ] Given any breach alert or breach log entry, no payload data, no `meal_text`, no usernames, no raw transcripts, no raw prompts, no raw media, no provider keys, and no provider responses appear (per ARCH-001@0.4.0 §10.7 emit-boundary redaction policy and §8.1 forbidden-field allowlist); only counts and tenant identifiers reduced to opaque IDs.
- [ ] Over any rolling 30-day window, the continuous detector catches every breach that the PRD-001@0.2.0 §6 K4 end-of-window audit (run via the dedicated audit role specified in ARCH-001@0.4.0 §9.2) catches: zero false-negatives.

### US-2: Automated model-stall detection (G2)
**As the Product Owner,** I want any LLM call that produces no token output for longer than a configurable threshold to be auto-detected and surfaced with takeover-trigger context, **so that** a stalled call does not silently exhaust time and budget while I am away from chat.
**Acceptance:**
- [ ] Given an LLM call in flight, when no token output is observed for ≥120 seconds (default; per-role configurable), then a stall event is emitted within ≤15 seconds of the threshold being crossed.
- [ ] Given a stall event, the Product Owner receives a Telegram alert containing the call identifier, the role, the elapsed time, and the prompt-token count, with no raw prompt text included (subject to the ARCH-001@0.4.0 §8.1 redaction policy).
- [ ] The detector covers ≥95 % of routed-model call paths and ≥95 % of direct-provider failover paths captured by the runtime.
- [ ] Synthetic stall scenarios injected at 120 s, 300 s, and 600 s elapsed all trigger detection within the ≤15-second target.

### US-4: Scale-ready Telegram allowlist with documented code-path audit (G4)
**As the Product Owner,** I want to grow the user base from the current 2 tenants up to thousands by editing a configuration file (no code change, no redeploy), with a documented and Reviewer-signed-off allowlist code-path audit before scaling past each load-test gate, **so that** I can scale-test the v0.2 product gradually (Product Owner → partner → friends → wider population) without coupling user growth to engineering cycles AND without risking a hidden code-path bottleneck (linear scan, slow store round-trip, missing cache layer).
**Acceptance:**
- [ ] Given a config-driven Telegram access list, when entries change from 2 to N, then no code deploy is required to enforce the new list.
- [ ] Given an updated access list, when configuration is committed, then access-control enforcement reflects the new list within ≤30 seconds.
- [ ] Before any allowlist load-test runs, the Architect publishes a written end-to-end audit of the allowlist code path (Telegram update arrival → access-check layer → cache layer if any → store round-trip if any → enforcement decision), explicitly enumerating every linear-scan, every store round-trip, every cache, and every fallback path; the Reviewer signs off on the audit document; sign-off is a blocking gate for load-tests.
- [ ] Allowlist load-tests pass at N = 2, N = 10, N = 100, N = 1 000, and N = 10 000 entries; passing each gate is a blocking pre-condition for advancing the user-count growth path past that gate.
- [ ] Per-update access-control overhead is ≤2 % of the user-facing latency budgets locked in PRD-001@0.2.0 §7 (voice ≤8 s p95, text ≤5 s p95, photo ≤12 s p95) at every load-test point.
- [ ] Removing a user from the access list immediately revokes access (no further user-scoped storage operations succeed for that `user_id` after revocation), and does not, by itself, delete that user's data; the PRD-001@0.2.0 §5 US-8 right-to-delete flow remains the only path that hard-deletes user data.

### US-5: Privacy-preserving telemetry (cross-cutting)
**As any user (Product Owner, partner, future user),** I want my data to remain redacted from every alert and structured-log channel that PRD-002 introduces, **so that** the new observability surface does not erode the data-isolation guarantee that PRD-001@0.2.0 §6 K4 promised.
**Acceptance:**
- [ ] Given any C10-emitted telemetry event introduced by PRD-002, no allowlisted forbidden field (`meal_text`, usernames, raw transcripts, raw prompts, raw media, provider keys, provider responses, per ARCH-001@0.4.0 §10.7 emit-boundary redaction policy and §8.1 allowlist) appears in alert or structured-log output channels.
- [ ] A sample audit of N ≥ 100 telemetry events drawn at random from the rolling-7-day window after v0.2 deploy shows 100 % redaction compliance.
- [ ] Any new telemetry event added by PRD-002 must declare which redaction-allowlist entries it asserts, and must be rejected at the emit boundary if it carries any non-allowlisted field.
- [ ] When a user invokes the PRD-001@0.2.0 §5 US-8 `/forget_me` flow, the right-to-delete transaction also hard-deletes that user's PRD-002 telemetry traces (breach-detection events tagged with that `user_id`, model-stall events scoped to that user's request, allowlist-revocation audit rows scoped to that user), in the same transaction boundary as the existing user-scoped data.
- [ ] PRD-002 telemetry that is subject to right-to-delete (any event tagged with a specific `user_id`) MUST be persisted exclusively in the durable tenant-scoped store (C3 tables, ARCH-001@0.4.0 §3.11) so that the §5 US-8 transaction is sufficient to delete it; any stdout-log emission of the same telemetry MUST redact `user_id` and every PII-bearing field at the emit boundary per ARCH-001@0.4.0 §10.7, so that ephemeral log retention windows (e.g. container-runtime stdout rotation) cannot become a deletion-gap that survives `/forget_me`.

## 7. Technical Envelope (constraints Architect must respect)
- **Channel.** Telegram remains the primary product channel as locked in PRD-001@0.2.0 §7 (Product-Owner-locked decision). PRD-002 introduces no new user-facing channel; web visualization is explicitly deferred (§3 NG4) to a successor PRD in the v0.2 roadmap. Operator-facing alerts introduced by PRD-002 are read by the Product Owner and the Architect (§4 P1, P4) and may be delivered through any operator-appropriate surface chosen by the Architect; they are not a user-facing channel.
- **Runtime.** openclaw runtime, locked in PRD-001@0.2.0 §7 and ARCH-001@0.4.0 (Product-Owner-locked decision). PRD-002 inherits this constraint without modification; the specific language and node version backing the runtime are Architect-owned and documented in ARCH-001@0.4.0.
- **VPS resource ceiling.** The PRD-001@0.2.0 §7 single-VPS ceiling (≤25 % CPU p95, ≤2 GB RAM steady) does NOT carry forward to v0.2 and beyond. PRD-002 sets no fixed CPU or RAM ceiling. The Architect must design for horizontal scale to ≥1 000 concurrent users (consistent with the §4 P3 growth path the Product Owner has confirmed: "это будет массовый продукт, разворачивать на тысячи юзеров"); concrete instance sizing is decided by the Architect and operations function on demand. Resource consumption should be proportional to active-user count, not unbounded.
- **LLM budget.** The PRD-001@0.2.0 §2 G5 / §6 K5 / §7 $10/month total cost ceiling does NOT carry forward to v0.2; the Product Owner has explicitly authorised unconstrained spend during the v0.2 development cycle ("на этапе разработки плевать на стоимость; мы вольны тратить столько, сколько потребуется для классного продукта"). PRD-002 commits no numeric LLM-budget envelope, neither for the SDLC pipeline (§3 NG9) nor for the product runtime (§9 OQ-1). Production-runtime cost telemetry for the assistant's own LLM calls is acknowledged as a future concern but is NOT a PRD-002 goal.
- **Latency budget.** PRD-002 telemetry collection MUST add ≤5 % overhead to every user-facing latency budget already locked in PRD-001@0.2.0 §7 (voice ≤8 s p95 / ≤30 s p100; text ≤5 s p95 / ≤15 s p100; photo ≤12 s p95 / ≤45 s p100). The allowlist access-control component additionally MUST add ≤2 % overhead at every load-test gate (G4 / K4). PRD-002 introduces two new operator-facing latency targets: K1 breach-detection p95 ≤5 min, K2 stall-detection emission ≤15 s after threshold cross.
- **Localization.** Russian-only user-facing UX, unchanged from PRD-001@0.2.0 §7. Operator-facing alerts may be in Russian or English at the Architect's discretion; user-facing surfaces (Telegram replies in PRD-001@0.2.0 §5 user stories) are not affected by PRD-002.
- **Compliance and PII handling.** PRD-002 telemetry channels (alerts, structured logs introduced by this PRD) MUST honour the redaction allowlist locked in ARCH-001@0.4.0 §8.1 and the emit-boundary redaction policy locked in ARCH-001@0.4.0 §10.7 (TKT-015@0.1.0 hardening). No new redaction-policy weakening is permitted by PRD-002. Telegram Bot API Terms of Service (`https://core.telegram.org/bots/terms`) compliance is unchanged from PRD-001@0.2.0 §7.
- **Data retention.** User-facing data-retention rules from PRD-001@0.2.0 §7 (raw audio and raw photo bytes deleted immediately after extraction; transcripts and confirmed meals retained until right-to-delete) carry forward unchanged. PRD-002 adds telemetry data (breach-detection events, model-stall events, allowlist-revocation audit rows) whose retention duration is open and tracked in §9 OQ-2.
- **Right-to-delete scope extension.** PRD-001@0.2.0 §5 US-8 right-to-delete is extended in PRD-002 §5 US-5 to also hard-delete the requesting user's PRD-002 telemetry traces (breach-detection events tagged with that `user_id`, model-stall events scoped to that user's request, allowlist-revocation audit rows scoped to that user) within the same transaction boundary as the existing user-scoped deletion in ARCH-001@0.4.0 §3.11. To make this transaction sufficient, PRD-002 telemetry subject to right-to-delete MUST be persisted exclusively in the durable tenant-scoped store (C3 tables, ARCH-001@0.4.0 §3.11); any stdout-log emission of the same telemetry MUST redact `user_id` and every PII-bearing field at the emit boundary per ARCH-001@0.4.0 §10.7, so that ephemeral log retention windows (e.g. container-runtime stdout rotation) cannot become a deletion-gap that survives `/forget_me`.
- **External dependencies.** Telegram Bot API; openclaw runtime; the LLM-router layer with direct-provider failover keys (per `docs/knowledge/llm-routing.md`). PRD-002 introduces no new external dependency. No web service, no external calendar provider, no new vendor lock-in is added by this PRD.

## 8. Risks & Mitigations
| Risk | Impact | Likelihood | Mitigation |
|---|---|---|---|
| **R1 — telemetry collection adds latency that breaches §7 user-facing budgets.** | high | medium | The Architect designs telemetry as asynchronous emit-and-forget against the existing C10 path; sampled metrics where full-volume telemetry would breach the ≤5 % overhead constraint; an explicit load-test gate in the implementation tickets blocks merge if the budget is violated. |
| **R3 — the breach detector becomes a cross-tenant query path itself.** | critical | low | The detector runs through the dedicated audit role gated by a separate runtime secret per ARCH-001@0.4.0 §9.1 / §9.2; it returns counts only, no payload; the Reviewer audits the detector implementation in the same way it audited the end-of-pilot K4 audit query. |
| **R4 — allowlist scaling reveals a per-update bottleneck (e.g. linear scan on every Telegram update, slow store round-trip per access check, missing cache layer).** | high | medium–high | The Architect publishes the documented end-to-end allowlist code-path audit (G4 acceptance) before any load-test runs; the Reviewer signs off on the audit; load-tests at N = 10 / 100 / 1 000 / 10 000 are gates in the implementation tickets, not after-the-fact validation; if any gate fails, the Architect redesigns before user growth past the last-passed gate is enabled. |
| **R5 — PRD-002 telemetry leaks PII (`meal_text`, usernames, raw transcripts, raw prompts, raw media) into alerts or structured logs.** | critical | low | PRD-002 inherits the ARCH-001@0.4.0 §10.7 emit-boundary redaction policy; US-5 acceptance includes a 100-event sample audit; any new event must declare its allowlisted fields and is rejected at the emit boundary if it carries non-allowlisted fields. |
| **R7 (weakest assumption A2) — continuous tenant-breach detection can run inside the existing C10 telemetry stack within the §7 telemetry-overhead constraint, without requiring a parallel detector pipeline.** | medium | medium | The Architect performs a feasibility study before committing detector design; if the constraint cannot be met inside C10, sampling fallback (≤1 in N persistence operations checked synchronously, the rest checked asynchronously in a parallel queue) is acceptable, with the sampling rate set to recover G1's zero-false-negative target on a rolling 30-day basis. |
| **R8 (weakest assumption A3) — Telegram allowlist scaling to ≥10 000 entries works through configuration rotation alone, with no deeper code-path bottleneck (e.g. Telegram update handler doing linear scan, per-message access check round-tripping a slow store, missing cache layer).** | medium–high | medium | Architect-published, Reviewer-signed allowlist code-path audit (G4 acceptance) is a blocking gate before any load-test. TKT-level load-tests at N = 10 / 100 / 1 000 / 10 000 gate merge; if a code-path bottleneck is found, a new caching component or store-backed lookup is required and the Product Owner is notified that the user-count growth path must pause at the last-passed gate until the bottleneck is resolved. |

The product-runtime cost envelope (the assistant's own LLM calls per user, per feature, per modality) remains TBD. Before the calendar-and-web-view PRD freezes its own §7, the Product Owner must ratify either a per-user $/month cap, a total $/month cap with tiered alerts, or an explicit "no enforced cap; track only" stance for production runtime. Default if unset at the PRD-002-merge moment: the existing PRD-001@0.2.0 §2 G5 / §6 K5 per-call cost telemetry already captured in v0.1 ARCH-001@0.4.0 §3.10 continues to emit, but no numeric ceiling is enforced and this remains a §9 OQ in the next PRD.
- **OQ-2 — telemetry data retention duration.** PRD-002 introduces breach-detection events, model-stall events, and allowlist-revocation audit rows whose retention duration is not specified. The Product Owner must ratify either a fixed window (e.g. 30 / 90 / 365 days) or a tiered policy (live store + cold archive + auto-purge). Default if unset before Architect handoff: 90-day live retention, then archive, then auto-purge at 365 days.
- **OQ-3 — numeric thresholds for K1 / K2.** The K1 breach-detection emission target latency in synchronous vs sampling-fallback configurations (per R7) and the K2 stall threshold (default 120 seconds; per-role configurable) require Architect feasibility input followed by Product Owner ratification. Default if unset before Architect handoff: K1 ≤5 min p95 in fully synchronous mode, ≤15 min p95 in sampling-fallback mode; K2 default 120 s, configurable per role between 60 s and 600 s.
- **OQ-4 — PRD-001@0.2.0 §3 NG10 status across the v0.2 roadmap.** PRD-002 preserves NG10 (no per-user assistant personality customization) without modification. The proactive-coaching PRD will introduce adaptive UX (assistant learning user preferences, asking adaptive questions, adjusting feature surfacing) but will not deprecate NG10 unless the Product Owner ratifies. The open question for the Product Owner is whether adaptive UX without preset selection counts as a NG10 deprecation in the proactive-coaching PRD; this is not a PRD-002 question per se but is recorded here so the Reviewer can verify it surfaces in the next PRD.
