# Q-TKT-043-01: GHCR image `ghcr.io/nicholasgriffintn/openclaw` does not resolve

## Context

TKT-043@0.1.0 ┬¦2 requires pinning `docker-compose.yml`'s `openclaw-gateway` service image (`ghcr.io/nicholasgriffintn/openclaw:latest`) to a `@sha256:<digest>` reference. TKT-043@0.1.0 ┬¦7 Constraints states: "Do NOT pin to a digest the executor cannot independently retrieve ŌĆö every digest must be reachable from a public registry at PR-merge time."

The executor attempted to retrieve the digest via three independent methods:
1. `docker manifest inspect ghcr.io/nicholasgriffintn/openclaw:latest` ŌåÆ `manifest unknown`
2. `docker buildx imagetools inspect ghcr.io/nicholasgriffintn/openclaw:latest` ŌåÆ `not found`
3. GHCR HTTP API (`/v2/nicholasgriffintn/openclaw/manifests/latest`) ŌåÆ `403 Forbidden` (anonymous) / `404 Not Found` (with `gh` token lacking `read:packages` scope)

All three return the same result: **the package does not exist or is private/inaccessible**.

Meanwhile, the upstream OpenClaw project's image at `ghcr.io/openclaw/openclaw:latest` **does** resolve and has a valid multi-arch index digest: `sha256:dcfd148777401d1bbdc63eab5c2f280bbfa912dfb1818566f9d66bb96ffb3f95`.

## Question

Which image path should the executor pin in `docker-compose.yml` for the `openclaw-gateway` service?

- **A)** Change to `ghcr.io/openclaw/openclaw@sha256:dcfd148777401d1bbdc63eab5c2f280bbfa912dfb1818566f9d66bb96ffb3f95` (the publicly available upstream image). The current `nicholasgriffintn/openclaw` path in docker-compose.yml appears to be stale/non-existent.
- **B)** Keep `ghcr.io/nicholasgriffintn/openclaw` and leave it un-pinned (tag-only), acknowledging that the image is private/not-yet-published and cannot be pinned per ┬¦7 Constraints. All other images get pinned.
- **C)** Keep `ghcr.io/nicholasgriffintn/openclaw` but the PO provides the digest out-of-band (e.g., from `docker inspect` on the VPS). The executor records it verbatim without independent verification ŌĆö an exception to ┬¦7 Constraints granted by the PO.

## What I assumed / would do if no answer

Option A ŌĆö change to `ghcr.io/openclaw/openclaw` and pin its digest. This is the only path where every `image:` reference can be pinned to a publicly retrievable digest, satisfying ┬¦7 Constraints and ┬¦6 AC #1 ("docker compose pull succeeds"). The current `nicholasgriffintn/openclaw` path returns `manifest unknown`, so any deploy with the current docker-compose.yml would also fail on `docker compose pull` for the gateway service.
