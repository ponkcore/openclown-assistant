# Q-TKT-031-01: Comment truncation limit (200 silent vs 280 with friendly notice)

## Context

Three upstream artefacts disagree on the mood-comment truncation contract:

- TKT-031@0.1.0 §2 In Scope, line 31: "Optional comment: truncate to ≤200 chars; drop comment if overlength rather than fail-open." (silent, 200-char limit)
- PRD-003@0.1.3 §2 G4 + §5 US-4: "1–10 + optional comment ≤280 chars" (280-char limit)
- ARCH-001@0.6.2 §6.2.2 C20 block: "Comment >280 chars: «Сократила комментарий до 280 символов. Записала настроение 7/10.»" (280-char limit + friendly notice)
- src/store/schema.sql `mood_events.comment_text CHECK (comment_text IS NULL OR length(comment_text) <= 280)` (matches PRD/ArchSpec)

The schema CHECK constraint and PRD/ArchSpec narrative agree on 280; the ticket §2 says 200 silent. Three things diverge: the limit (200 vs 280), the on-overflow behaviour (silent drop vs friendly notice), and the operation (truncate vs drop-entirely). The ticket §2 phrase "truncate to ≤200 chars; drop comment if overlength rather than fail-open" is itself ambiguous between "truncate to first 200 chars" and "drop the entire comment when overlength".

## Question

Pick one canonical contract for mood-comment overflow handling. Multiple-choice:

- A) §6.2.2 contract: limit=280, behaviour=truncate-to-280-chars, reply="Сократила комментарий до 280 символов. Записала настроение N/10." (matches PRD + ArchSpec + schema; supersedes ticket §2 200-silent statement as a ticket typo).
- B) Ticket §2 contract as written: limit=200, behaviour=drop-overflow-entirely, reply=silent (no overflow notice; success reply unchanged); requires the schema's 280 CHECK to be a defensive ceiling above the application's 200 limit.
- C) Hybrid: limit=200 enforced silently in app layer, 280 schema CHECK as belt-and-braces, no friendly notice; this is the iter-1 Executor implementation.

## What I assumed / would do if no answer

Iter-1 implementation followed C (200 silent) per the ticket §2 explicit statement. Iter-2 has not changed this. Q-file filed for PO/Architect arbitration; merge proceeds with C until answered. If the answer turns out to be A (the §6.2.2 path), this is a one-line copy + one-line truncation-policy change in src/modality/mood/logger.ts plus a string addition in src/modality/mood/copy.ru.ts plus updated tests — small follow-up TKT scope.

## Architect's answer

**Pick: Option A — `limit=280`, `behaviour=truncate-to-280-chars`, `reply="Сократила
комментарий до 280 символов. Записала настроение N/10."`** Aligns with PRD-003@0.1.3
§2 G4 + §5 US-4 ("optional comment ≤280 chars"), with ARCH-001@0.6.2 §6.2.2 C20 verbatim
copy, and with the deployed `mood_events.comment_text CHECK (length(comment_text) <=
280)` schema constraint already on `main`.

**Why not C (the current shipped code, 200 silent):**

- Three of four upstream artefacts (PRD, ArchSpec, schema) agree on 280 with a friendly
  notice. The TKT-031@0.1.0 §2 In-Scope statement of "200 silent" is the outlier; the
  ticket text drift is the typo, not the PRD / ArchSpec / schema cluster.
- Silent drop loses user-meaningful content without telling the user, which contradicts
  ARCH-001@0.6.2 §6.2.1 tone profile "when extraction is uncertain, ASK — do not guess"
  applied to the truncation case ("when input is over the limit, TELL — do not silently
  drop").
- Truncate-not-drop preserves the first 280 chars of user thought; drop-overflow
  preserves zero. Truncation is the higher-fidelity option for a tracking surface.

**Why not B (200 limit + drop-entirely + silent):**

- Same fidelity argument as C, plus the `200` limit doesn't match any other artefact in
  the repo. It would be a third truncation contract introduced just for this code path.

**Required follow-up:** A new TKT-047@0.1.0 aligns the shipped code (`src/modality/mood/`)
to Option A — `200` → `280`, drop-overflow → truncate-to-280, silent → friendly Russian
notice. The TKT-031@0.1.0 §2 In-Scope text is also updated by the same executor cycle to
say "Optional comment: truncate to ≤280 chars; emit friendly Russian notice on overflow"
to remove the upstream drift; the schema CHECK and ArchSpec / PRD strings are already
correct.

This question is now `status: answered`. The follow-up ticket is TKT-047@0.1.0 (created
in the same ARCH-001@0.7.0 PR as this answer).
