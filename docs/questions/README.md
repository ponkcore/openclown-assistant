# Questions (Q-TKT)

Owner: **executor** (asks) + **Technical Architect** (answers).

## Rules

- An Executor MUST stop coding and create a Q-TKT when:
  - The Ticket is genuinely ambiguous (two reasonable readings differ).
  - The Ticket contradicts the ArchSpec or an ADR.
  - The Ticket asks for something physically impossible within the Technical Envelope.
  - The Executor needs to add a dependency not in §7 Constraints allowlist.
- Filename: `Q-TKT-XXX-NN.md` (auto-generated by `python scripts/new_artifact.py question "TKT-XXX <topic>"`).
- The Ticket's frontmatter `status` flips to `blocked` until the Architect answers.
- Prefer **binary or multiple-choice** questions over open-ended ones — one decisive answer.
- An Executor who guesses instead of asking is failing their role.

## Lifecycle

`open` → `answered` (Architect commits the answer) → Executor resumes; Ticket flips back to `in_progress`.
`superseded` if the Ticket itself is rewritten.
